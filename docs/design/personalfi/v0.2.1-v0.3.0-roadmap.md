# PersonalFi — v0.2.1 → v0.3.0 Roadmap

> **Date:** 30 March 2026 | **Status:** Draft  
> **Baseline:** v0.2.0 — 355 tests, 55 endpoints, 13 views, 13 route modules  
> **Scope:** 10 iterations (v0.2.1 through v0.3.0), one sprint each

---

## 1. Expert Panel Findings

### 1.1 UI/UX Designer

**Critical gaps:**

- **No data visualization.** Every competitor has charts. PersonalFi renders reports as text tables. Users cannot grasp spending trends at a glance. This is the single biggest UX gap versus any competitor.
- **Tags exist in the schema but have zero frontend surface.** The `tags` column on transactions stores JSON but there's no UI to create, assign, or filter by tags. This is dead schema — confusing if users discover the field in exports.
- **No search.** 13 views but no global search. Users with 100+ transactions have no fast way to find "that Uber charge from February." Every competitor has search.
- **Recurring transactions are invisible.** The scheduler spawns them silently. Users cannot see upcoming bills, edit rules, or pause a rule from the UI. Subscriptions view partially covers this, but recurring rules are a separate system with no frontend.
- **Settings view is thin.** No password change, no account deletion, no theme switching. Users who want to change their password must… they can't.
- **Split UX is limited to equal/exact.** The schema supports `percentage` and `shares` but the service only implements `equal` and `exact`. Friend groups splitting a dinner 60/40 can't do it.

**Delighters:**

- Budget progress bars with color thresholds (green → yellow → red)
- Spending sparklines in the accounts list
- "Upcoming this week" card on dashboard (combines recurring + subscription due dates)
- Keyboard shortcut hints overlay (discoverability for existing shortcuts)

### 1.2 Software Architect

**Technical debt:**

- **No repository layer.** Routes contain raw SQL queries mixed with business logic. 13 route files doing direct `db.prepare()` calls. The `src/repositories/` directory exists and is empty. This makes testing harder, introduces SQL duplication, and tangles concerns.
- **Duplicated CSRF export.** `src/middleware/csrf.js` exports `createCsrfMiddleware` twice (line 53 and 55). Harmless but sloppy.
- **No input validation on auth routes.** `src/routes/auth.js` does manual `if (!username || !password)` checks instead of using the Zod `validate()` middleware that every other route uses. Inconsistency.
- **Scheduler is tightly coupled.** `spawnDueRecurring()` directly mutates account balances and inserts transactions inside the scheduler module. Should delegate to a transaction service.
- **No migration versioning beyond 001.** Only one migration file exists. Future schema changes need a proper migration pipeline — the foundation is there but untested at scale.
- **`config.js` reads version from `package.json` on every require.** Minor, but `Object.freeze` on the config means this is fine — just unusual.

**Infrastructure improvements needed:**

- API versioning (even just `/api/v1/` prefix) before adding more endpoints
- Request ID tracking for correlating logs
- Graceful shutdown is configured but has no integration test
- Health check endpoint (`/api/health`) for monitoring/uptime tools

**Edge cases missed:**

- Transaction with amount = 0
- Account balance going deeply negative (no enforcement)
- Subscription with past `next_billing_date`
- Category deletion when budget items reference it
- Group deletion cascade behavior on unsettled expenses

### 1.4 Product Manager

**MVP gap analysis — what's missing for "daily driver" status:**

1. **Recurring transactions UI** — users can't manage their bills. Subscriptions view is a partial workaround but the recurring rules system is backend-only. This is the #1 gap preventing daily use.
2. **Charts** — text reports aren't engaging enough for daily check-ins. Users open the app, see numbers, close it. A chart makes them *explore*.
3. **Search** — once you have 3+ months of data, you need search. Without it, the app doesn't scale with usage.
4. **Password management** — basic account security. Users will abandon an app they can't secure.
5. **Tags** — the flexible categorization layer that power users expect. Categories are hierarchical and structured; tags are freeform and cross-cutting. Both are needed.

**Retention drivers (in priority order):**

1. Visual feedback on financial progress (charts, progress bars)
2. "Set and forget" automation (recurring transactions, budget rollover)
3. Collaborative features that reduce friction (percentage splits, shared budgets)
4. Data confidence (net worth tracking over time, audit visibility)

### 1.5 Sales/Marketing Analyst

**Competitive positioning:**

PersonalFi's unique value is **self-hosted + collaborative** — no competitor combines both. But the current feature set doesn't yet *demonstrate* this advantage:

- Splitwise users switching to PersonalFi lose percentage/shares splits
- YNAB users switching lose budget rollover and net worth tracking
- Actual Budget users switching lose charts/graphs
- Firefly III users switching lose multi-currency and tags

**Features that would make PersonalFi competitive:**

| Feature | Competitive Impact | Effort |
|---------|-------------------|--------|
| Charts/graphs | Table stakes — every competitor has them | Medium |
| Net worth tracking | YNAB's killer feature, Firefly III has it | Small |
| Budget rollover | YNAB's envelope method depends on this | Small |
| Percentage/shares splits | Splitwise parity | Small |
| Tags | Firefly III / GnuCash parity | Small |
| Multi-currency | Firefly III's differentiator | Large (defer) |
| Search | Table stakes | Medium |

**Self-hosted advantage to emphasize:**

- No subscription fee (vs YNAB $99/yr, Mint was ad-supported)
- Data ownership (Mint shut down, moved users to Credit Karma)
- Collaborate without sharing data with a third party

### 1.6 Security Engineer

**Critical findings:**

1. **No password change endpoint.** Users cannot rotate compromised credentials. This is a security requirement, not a feature request.
2. **No account deletion.** GDPR/privacy compliance gap. Users should be able to delete their data.
3. **Session tokens are random hex but never rotated.** After login, the same token is used for up to 30 days. Post-authentication session fixation risk. Should rotate token on sensitive operations.
4. **No rate limiting on password verification.** The auth rate limiter exists in config but the `authLimitMax` of 20 attempts per 15 minutes is generous. Brute force is feasible for weak passwords.
5. **Audit log exists but has no viewer.** Logs are written but only purgeable. No API to read them. If an account is compromised, the user can't review what happened.
6. **No Content-Security-Policy header.** Helmet is configured but CSP isn't mentioned. Vanilla JS SPA should have a strict CSP.
7. **bcrypt salt rounds = 12.** Acceptable but could document the reasoning. Cost factor 12 is \~250ms per hash, which is fine.
8. **Export endpoint returns all user data as JSON.** No re-authentication required. If a session is hijacked, all financial data can be exfiltrated in one call.

**Recommended priority:**

1. Password change (v0.2.1) — highest security impact
2. Audit log viewer (v0.2.x) — incident response capability
3. Session rotation on password change
4. CSP headers
5. Re-auth for data export (can defer)

## 3. Prioritized Iteration Plan

### Prioritization rationale

The ordering follows this framework:
1. **Security first** — password change and account security
2. **Complete half-built features** — recurring UI, tags, percentage splits (schema exists, just needs surface)
3. **Retention drivers** — charts, search, net worth
4. **Competitive parity** — budget rollover, shared budgets, audit viewer
5. **Polish** — each iteration leaves the app shippable

## 5. Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| **Chart.js bundle size bloats PWA** | Medium | Medium | Use CDN with async loading; measure before/after service worker cache size; consider lighter alternative (uPlot ~35KB) if too heavy |
| **Tag migration breaks existing transaction data** | Medium | High | Migration adds join table alongside existing JSON `tags` column; migrate data in migration script; keep JSON column as fallback for one version |
| **Budget rollover calculation complexity** | Medium | Medium | Start with monthly-only rollover; defer quarterly/yearly rollover edge cases to v0.3.x; comprehensive TDD with period boundary tests |
| **Shared budget authorization complexity** | Medium | Medium | Reuse existing group role checks (owner/admin can manage); members can only view; test all role permutations |
| **Scheduler reliability for net worth snapshots** | Low | Low | Snapshots are idempotent (UNIQUE on user_id + date); missed snapshot can be manually triggered; catch-up logic on server restart |
| **Search performance with large datasets** | Low | Medium | SQLite FTS5 available if LIKE queries become slow; start with LIKE, measure at 10K+ transactions; add FTS5 migration if needed |
| **Scope creep per iteration** | Medium | High | Each iteration spec is locked before implementation; unfinished work moves to next iteration, never stretches current one |
| **Breaking API changes** | Low | High | All new endpoints are additive; no existing endpoint signatures change; response shapes extend (add fields), never remove |
| **Password change invalidates active sessions** | Intended | N/A | This is by design — intentional session rotation on password change; document in API reference |

## 7. Out of Scope (v0.3.x+)

These features were considered but deferred beyond v0.3.0:

- **Multi-currency conversion** — requires exchange rate API, significant complexity
- **Two-factor authentication** — important but not blocking for self-hosted single-user/household
- **Attachments/receipts** — needs file storage strategy (filesystem vs blob)
- **Export to PDF** — low demand, can use browser print as workaround
- **Theme switching** — Midnight theme is solid; light theme is cosmetic, not functional
- **Mobile native app** — PWA covers mobile; native is a separate project
- **Bank API integration** — Plaid/Open Banking is a v1.0+ initiative
- **AI-powered categorization** — current rule-based system works; ML is over-engineering for v0.3
