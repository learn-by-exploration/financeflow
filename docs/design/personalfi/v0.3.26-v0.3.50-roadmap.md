# PersonalFi — v0.3.26 → v0.3.50 Roadmap

> **Created:** 30 March 2026 | **Current:** v0.3.25 (948 tests) | **Target:** v0.3.50
> **Scope:** 25 iterations across 3 phases

---

## Expert Panel Audit

### 1. UI/UX Designer — Audit

**Gaps identified:**
- **No frontend views for 40%+ of backend features.** Charts, insights, reports, notifications, attachments, search, net worth, reminders, recurring suggestions, duplicates — all have API endpoints but NO dedicated UI views. Users can't access these features.
- **No responsive table layouts.** Transaction tables break on mobile (<768px).
- **No loading states or skeleton screens.** Views flash empty before data loads.
- **No form validation feedback.** Client-side validation relies entirely on server 400s — no inline field errors.
- **No pagination controls in UI.** Backend supports pagination but the frontend renders only first page of most lists.
- **No data visualization in dashboard.** Chart.js is loaded via CDN but the dashboard renders static HTML cards, not actual charts.
- **Accessibility gaps.** No ARIA labels on interactive elements, no focus management on view transitions, no skip-to-content link, keyboard navigation is limited to modal shortcuts.
- **No multi-currency display in UI.** Backend supports per-account currency but UI hardcodes formatting.
- **No empty state illustrations.** Empty lists show nothing — no guidance for new users.

**Proposals (prioritized):**
- CRITICAL: Frontend views for charts, insights, notifications, search results
- CRITICAL: Responsive tables with card layout on mobile
- HIGH: Loading skeletons, error states, empty states
- HIGH: Client-side form validation with inline errors
- HIGH: Pagination controls component
- MEDIUM: Dashboard chart integration (Chart.js rendering, not just data)
- MEDIUM: Accessibility audit — ARIA, focus management, screen reader
- LOW: Theme customization beyond dark mode

---

### 2. Software Architect — Audit

**Gaps identified:**
- **No service layer for most domains.** Only `transaction.service.js`, `split.service.js`, `notification.service.js`, `health.service.js`, `recurring-detector.js` exist. Routes like accounts, budgets, goals, categories, subscriptions have business logic directly in route handlers.
- **No group/split repository.** Groups and splits hit DB directly in route handlers — no repository layer.
- **No input validation on group/split routes.** Zod schemas exist for most entities but groups/splits/settings/admin routes lack schema validation.
- **Missing Zod schemas:** `group.schema.js`, `split.schema.js`, `settings.schema.js`, `search.schema.js`, `notification.schema.js`, `attachment.schema.js`, `report.schema.js`, `chart.schema.js`, `admin.schema.js`
- **No database connection pooling strategy.** Single `db` instance — fine for SQLite but no connection health monitoring.
- **No request timeout middleware.** Long-running queries can hang indefinitely.
- **No API versioning.** All routes on `/api/` with no version prefix — future breaking changes have no migration path.
- **No structured logging correlation.** Request IDs exist but log entries don't consistently include them.
- **Cache invalidation is missing.** In-memory cache exists but there's no invalidation on writes — stale data on reports/insights after mutations.

**Proposals (prioritized):**
- CRITICAL: Missing Zod schemas for unvalidated endpoints
- HIGH: Service layer extraction for accounts, budgets, goals, subscriptions
- HIGH: Group/split repository extraction
- HIGH: Cache invalidation on write operations
- MEDIUM: Request timeout middleware
- MEDIUM: API versioning strategy (v1 prefix)
- LOW: Structured logging improvements

### 3. QA Engineer — Audit

**Gaps identified:**
- **No frontend tests.** 948 tests are all backend API tests. Zero DOM/UI testing.
- **No load/stress testing.** Performance test file exists but tests response time of individual endpoints, not concurrent load.
- **No test for scheduler edge cases.** What happens when scheduler spawns during a request? Race conditions between scheduler and API writes.
- **No test for attachment file cleanup.** Deleting a transaction with attachments — are files removed from disk?
- **No test for backup/restore roundtrip.** Backup is tested but full restore-and-verify isn't.
- **No test for concurrent user sessions.** Multi-session behavior, session revocation while requests are in-flight.
- **No test for exchange rate impact on reports.** Multi-currency transactions calculated against exchange rates in reports/insights.
- **No test for large dataset performance.** How does the system behave with 10K+ transactions, 50+ accounts?
- **No mutation/fuzzing tests.** No boundary testing beyond basic validation (e.g., extremely long strings, Unicode edge cases, null bytes).
- **Group authorization edge cases untested.** Guest member permissions, role changes during active splits, owner transfer.

**Proposals (prioritized):**
- CRITICAL: Stress tests — 10K+ transactions, concurrent operations
- HIGH: Scheduler concurrency and edge case tests
- HIGH: Attachment lifecycle tests (upload → query → delete → verify cleanup)
- HIGH: Backup/restore roundtrip verification
- MEDIUM: Fuzzing/boundary tests on all string inputs
- MEDIUM: Multi-currency report accuracy tests
- LOW: Frontend smoke tests (if frontend gets more complex)

### 6. Security Engineer — Audit

**Gaps identified:**
- **No password reset flow.** If user forgets password, only option is DB-level intervention. Self-hosted still needs admin-assisted reset.
- **No 2FA/TOTP support.** Single-factor auth only — password compromise = full account compromise.
- **No API rate limiting per user.** Global rate limiter exists but no per-user throttling — a compromised session can hammer the API.
- **Session tokens stored in plain text in DB.** Should be hashed — DB leak reveals valid sessions.
- **No Content Security Policy reporting.** CSP is set via Helmet but no `report-uri` — violations are silent.
- **No audit log retention policy.** Audit log grows unbounded — potential storage DoS.
- **Attachment uploads lack virus scanning.** Multer accepts files but no content verification beyond MIME type.
- **No CORS origin configuration in .env.** CORS is restrictive but hardcoded — should be configurable per deployment.
- **API tokens have no expiry.** Once created, valid forever until manually revoked.
- **No failed login notification.** Account lockout exists but no notification to the account owner.
- **Backup files not encrypted.** Database backups stored as plain SQLite files — contain passwords (hashed) and all financial data.

**Proposals (prioritized):**
- CRITICAL: Session token hashing in database
- CRITICAL: API token expiry with rotation
- HIGH: Admin-assisted password reset flow
- HIGH: Two-factor authentication (TOTP)
- HIGH: Per-user rate limiting
- HIGH: Audit log retention with auto-cleanup
- MEDIUM: Backup encryption
- MEDIUM: CORS origin from environment variable
- MEDIUM: CSP violation reporting endpoint
- LOW: Attachment content verification

#### v0.3.27 — Session Token Hashing & API Token Expiry

**Why:** Plaintext session tokens in DB mean a DB leak compromises all active sessions. API tokens without expiry are a ticking liability.

**Scope:**
- Hash session tokens with SHA-256 before DB storage; compare on lookup
- Add `expires_at` column to `api_tokens` table (migration 013)
- Reject expired API tokens in auth middleware
- Add `PUT /api/tokens/:id/rotate` endpoint — issue new token, revoke old
- Add token expiry parameter to `POST /api/tokens`

**Files to create:**
- `src/db/migrations/013_token_security.sql`

**Files to modify:**
- `src/middleware/auth.js`
- `src/routes/auth.js`
- `src/routes/api-tokens.js`
- `src/schemas/api-token.schema.js`

- Add `X-Cache` response header (HIT/MISS) for debugging

#### v0.3.36 — Admin Password Reset & Two-Factor Auth (TOTP)

**Why:** No password recovery mechanism locks users out permanently. TOTP is table-stakes security for finance apps.

**Scope:**
- Admin-assisted password reset: `POST /api/admin/users/:id/reset-password` generates a temporary password (admin-only endpoint)
- User-initiated password reset flow for self-hosted: admin configurable, logged to audit
- TOTP setup: `POST /api/auth/totp/setup` returns QR code (base32 secret)
- TOTP verify: `POST /api/auth/totp/verify` enables 2FA after successful code validation
- TOTP on login: if 2FA enabled, login returns `{ requires_2fa: true, session_token: partial }`, second step `POST /api/auth/totp/login` completes auth
- TOTP disable: `DELETE /api/auth/totp` with password confirmation
- Backup codes (8 one-time codes) generated on TOTP setup

**Files to create:**

#### v0.3.44 — Accessibility Compliance (WCAG 2.1 AA)

**Why:** Financial apps have legal and ethical obligations to be accessible. Current frontend has no ARIA labels, poor keyboard navigation, and no focus management.

**Scope:**

- Keyboard navigation: Tab order through all views, Enter/Space to activate, Escape to close
- Color contrast audit: ensure all text meets 4.5:1 ratio against dark theme backgrounds
- Screen reader announcements for dynamic content (toast notifications, view changes, loading states)
- `prefers-reduced-motion` media query: disable animations for users who prefer reduced motion
- Add `role`, `aria-live`, `aria-label`, `aria-describedby` attributes throughout

#### v0.3.45 — Backup Encryption & CORS Configuration

**Why:** Backups contain all financial data in plaintext SQLite. CORS is hardcoded — deployers need to configure allowed origins.

**Scope:**

- `POST /api/admin/backup` accepts optional `passphrase` parameter
- Encrypted backups have `.enc` extension; download/restore detects format
- `POST /api/admin/restore` accepts `passphrase` for encrypted backups
- CORS origin from `CORS_ORIGIN` environment variable (comma-separated list)
- Default CORS: same-origin only (current behavior preserved)
- CSP `report-uri` endpoint: `POST /api/admin/csp-report` logs violations

**Files to modify:**
