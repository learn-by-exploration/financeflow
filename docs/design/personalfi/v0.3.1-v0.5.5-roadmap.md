## 1. Expert Panel Findings

### 1.1 UI/UX Designer

**Missing user flows:**
- **No bulk operations** — Users can't select and delete/categorize/tag multiple transactions at once. High-volume users (CSV importers) are stuck editing one-by-one.
- **No transaction duplication** — "Repeat this transaction" is a common need (similar receipts), forces re-entry of all fields.
- **No undo after delete** — Destructive actions are instant with no recovery window. Confirm dialogs exist but no soft-delete/trash.
- **No date navigation** — Calendar/month picker for jumping through transaction history. Users scroll endlessly.
- **No empty states** — New features (budgets, goals, groups) show blank screens with no guidance on what to do.
- **Filter state is lost** — Navigating away from transactions resets all filters. No deep-linking/URL state.

**Polish needed:**
- Loading skeletons — the app shows nothing during API calls, then pops in. Jarring on slow connections.
- Toast notifications stack on top of each other, no queuing.
- Mobile responsiveness is basic — sidebar hamburger exists but forms/modals overflow on small screens.
- No keyboard navigation within modals (Tab order, focus trapping).

**Accessibility gaps:**
- No ARIA labels on icon-only buttons (edit/delete pencil/trash icons).
- Color-only indicators (red/yellow/green budget progress) — fails contrast for color-blind users. Need text/icon fallbacks.
- No skip-to-content link.
- Chart.js charts have no `aria-label` or text alternative.
- Focus management after modal close drops focus to top of page.

### 1.2 Software Architect

**Technical debt:**
- **No repository layer** — `src/repositories/` is empty. Route files contain raw SQL queries inline. This violates the documented architecture (routes → services → repositories). Some routes are 200+ lines of mixed SQL and HTTP logic.
- **No Zod schemas** — `src/schemas/` is empty. Validation is done inline in routes with manual `if (!name)` checks. The spec mandates Zod.
- **Inconsistent error responses** — Some routes return `{ error: 'message' }`, others `{ error: { code, message } }`. No standard envelope.
- **No pagination on list endpoints** — `GET /api/transactions` supports it, but accounts, categories, subscriptions, goals, tags do not. Will break with scale.
- **CSRF middleware exported twice** — `module.exports = createCsrfMiddleware;` appears twice at EOF.
- **No graceful degradation on scheduler** — If recurring rule references a deleted account, the scheduler crashes.
- **`cors()` with no origin restriction** — Accepts requests from any origin. Fine for self-hosted dev, dangerous if exposed.
- **No request body size limit per route** — Global 1MB limit is set, but data import should allow larger payloads while auth should be smaller.
- **Transaction tags stored as JSON string** — `tags TEXT DEFAULT '[]'` in transactions table, but there's also a `transaction_tags` join table from migration 002. Dual storage is confusing.
- **No database connection pooling concern** — better-sqlite3 is single-connection, synchronous. Fine for low concurrency but worth documenting the limitation.

**Performance concerns:**
- No indexes on `shared_expenses(group_id)`, `expense_splits(expense_id)`, `settlements(group_id)`.
- Financial health score recalculates from scratch on every request — no caching.
- Dashboard stats query hits multiple tables with no materialized view or cache.
- Budget vs actual comparison does a full table scan on transactions for the date range.

### 1.3 QA Engineer

**Untested edge cases:**
- What happens when a user creates a transfer between two accounts they own, then deletes one account? The cascade should clean up, but the paired transaction's account reference becomes dangling.
- Budget rollover across year boundaries (December → January) — does the period math work?
- CSV import with malformed dates, negative amounts, Unicode descriptions, BOM headers.
- Session expiration during a multi-step operation (e.g., creating a group expense mid-flow).
- Concurrent user modifications to the same group expense.
- Scheduler running while a user simultaneously deletes recurring rules.
- What happens when `balance` overflows REAL precision? (amounts > 2^53).
- Tags with special characters (quotes, HTML, emojis) in name field.
- Financial health score when user has only income (no expenses) or only expenses (no income).
- Net worth snapshot when all accounts have `include_in_net_worth = 0`.

**Missing integration tests:**
- Full lifecycle: register → create accounts → import CSV → verify balances → create budget → check budget vs actual → export → re-import → verify data integrity.
- Multi-user collaboration: User A creates group → User B joins → both add expenses → verify balances → settle → verify zero balances.
- Scheduler integration: Create recurring rule → advance time → verify transactions spawned → deactivate rule → verify no more spawns.
- Data export/import round-trip: export all data → delete user → re-register → import → verify everything matches.

**What could break:**
- SQLite REAL for currency amounts causes floating point drift. `0.1 + 0.2 !== 0.3`. Not using integer cents.
- `datetime('now')` in SQL defaults to UTC but frontend may display local time. Timezone mismatches in date filtering.
- Password hash timing attacks — bcrypt.compare is constant-time, but returning early on "user not found" vs "wrong password" leaks user existence.

### 1.4 Product Manager

**Features that would drive adoption:**
1. **Recurring transaction auto-detection** — "You've entered 'Rent' 3 months in a row on the 1st. Create a recurring rule?" This is a killer engagement driver.
2. **Bill reminders / upcoming expenses view** — Show what's due this week. Subscriptions know next billing date; surface it proactively.
3. **Year-in-review / monthly summary report** — Automated digest. "You spent ₹45,000 in March, 12% more than February. Top increase: Dining (+₹3,200)."
4. **Multi-currency support** — Deferred to v2 in spec, but even basic display (₹, $, €) without conversion would serve international users.
5. **Attachments/receipts** — Photo of receipt linked to transaction. Huge for expense tracking discipline.
6. **API tokens for automation** — Power users want to script imports, build integrations. Personal API keys are low-cost, high-value.

**Competitive gap vs Monarch/YNAB/Splitwise:**
- See Section 2 below.

### 1.5 Sales/Marketing

**Marketable features (what makes people share it):**
- **"One app for everything"** — Budgeting + splitting + subscriptions + health score. No competitor does all four.
- **Self-hosted / privacy** — Post-Mint shutdown, privacy is a selling point. "Your data never leaves your server."
- **No subscription fee** — Monarch is $99/year, YNAB is $99/year. PersonalFi is free.
- **Couples/roommate focus** — Splitwise doesn't do budgeting. YNAB doesn't do splitting. PersonalFi does both.

**"Wow factor" needed:**
- **Visual dashboards that look premium** — The current Chart.js sparkline is basic. Need category donuts, trend lines, comparison charts.
- **One-click insights** — "You're overspending on dining by ₹3,200 vs last month" as a card on the dashboard.
- **Shareable reports** — Generate a PDF or image summary for a trip group or monthly budget.
- **Quick setup** — Docker one-liner that works. `docker compose up` and you're running.

### 1.6 Security Engineer

**Vulnerabilities / hardening needed:**

| Priority | Issue | Risk | Fix |
|----------|-------|------|-----|
| **HIGH** | `cors()` with no origin restriction | CSRF bypass from any domain | Configure `origin` to same-host or configurable allowlist |
| **HIGH** | No session invalidation on password change (partially done — need to verify all sessions are killed) | Session hijacking after credential change | Invalidate ALL sessions except current on password change |
| **HIGH** | No account lockout after failed login attempts | Brute force attacks | Lock after N failures with exponential backoff |
| **MEDIUM** | Rate limiter disabled in test mode entirely | Can't test rate limiting behavior | Use higher limits in test, not disabled |
| **MEDIUM** | CSRF token in cookie without `HttpOnly` flag | XSS can read CSRF token | This is by design (double-submit pattern reads from cookie via JS), but document this |
| **MEDIUM** | No Content-Type validation on POST/PUT | Request smuggling edge cases | Reject non-JSON Content-Type on API routes |
| **MEDIUM** | Audit log doesn't capture IP address or user-agent | Insufficient forensics | Add `ip` and `user_agent` columns |
| **LOW** | No session rotation on privilege escalation | Session fixation (low risk for self-hosted) | Rotate token on password change (already done) |
| **LOW** | No password complexity requirements | Weak passwords | Add minimum length + complexity rules |
| **LOW** | `x-session-token` header auth has no expiry check on the header itself | Token enumeration (low risk) | Already checked via DB expiry |
| **LOW** | No CSP reporting endpoint | Can't detect XSS attempts | Add `report-uri` directive |

**OWASP Top 10 compliance:**
- **A01 Broken Access Control** — Mostly covered (ownership checks), but shared budget items don't verify the user is a group member in all paths.
- **A02 Cryptographic Failures** — bcrypt is good. Session tokens use crypto.randomBytes. OK.
- **A03 Injection** — Parameterized queries throughout. OK.
- **A04 Insecure Design** — No rate limiting on auth endpoints specifically (global rate limit exists but is generous).
- **A05 Security Misconfiguration** — CORS is wide open. Helmet is configured well otherwise.
- **A07 Auth Failures** — No account lockout, no password complexity.
- **A09 Logging & Monitoring** — Audit log exists but no alerting, no IP tracking.

#### v0.4.1 — Spending Insights & Anomaly Detection

**Theme:** Proactive intelligence — detect unusual patterns.

**Scope:**
- New API: `GET /api/insights` — returns actionable insights based on transaction history:
  - **Unusual spending**: "You spent ₹12,000 on Electronics this month — 3× your average"
  - **Recurring pattern detection**: "You've paid ₹499 to Netflix on the 15th for 3 months. Create a subscription?"
  - **Category budget breach prediction**: "At current pace, you'll exceed your Dining budget by ₹1,200"
  - **Subscription overlap**: "You have both Spotify and Apple Music. Consider canceling one?"
- Algorithm: Compare current month spending per category against 3-month rolling average. Flag >50% deviations.
- Recurring detection: Find transactions with same description + similar amount (±10%) recurring at regular intervals (28-31 days for monthly)
- Store insights in new `insights` table with `dismissed_at` column so users can dismiss them

**New tests (10):** `tests/insights.test.js`
- Anomaly detected when spending 2× average
- No anomaly for categories with < 3 months history
- Recurring pattern detected for 3+ occurrences
- Budget breach prediction math is correct
- Dismissed insights don't reappear
